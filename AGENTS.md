# GridStart 开发约束

## 官方资料核验门禁

- 涉及框架、操作系统 API、第三方库、模型能力、参数、版本兼容性或平台限制时，禁止凭记忆直接下结论。
- 回答、设计或修改代码前，必须核对与项目实际版本匹配的官方文档、官方源码或官方示例。
- 技术资料优先级必须是：项目当前代码、`Cargo.lock` 和本机依赖源码 > 对应版本的官方 API 文档 > 官方仓库源码、示例和变更记录 > 其他辅助资料。
- 找不到官方依据时，必须明确说明“尚未确认”，不得自行补全参数、组件、行为或限制。
- 不得混淆官方 API 参数、官方返回字段、应用自行实现的后处理规则和产品界面设置。
- 声称某个参数、组件或能力存在时，必须能够指出准确的官方类、方法、属性或组件名称。
- 官方存在新旧两套 API 时，必须确认项目当前使用哪一套、新 API 的系统版本、硬件、运行时和打包要求，以及当前设备能否实际验证。
- 核验尚未完成时必须停止实现，不得先写猜测版本或隐式回退路径。

## 实施前核验记录

- 开始实现外部 API 或 UI 能力前，必须先向用户简短说明：当前依赖或 API 版本、已核对的官方资料、准备使用的官方组件或方法、已确认的限制、需要自定义实现的原因。
- 修改 Slint UI 前必须列出已检查的 Slint 原生组件名称；决定自定义实现时，必须记录原生组件不能满足的具体行为。
- 没有完成上述核验记录，不得创建自定义 `Window`、弹层、菜单、选项卡、焦点管理逻辑，也不得接入新的平台 API。

## Slint 组件优先级

- 新增或修改 UI 功能前，先确认 `Cargo.lock` 中实际使用的 Slint 版本。
- 先检查对应版本的 Slint 官方文档；必要时检查本机 Cargo registry 中该版本的 `builtins.slint`、标准组件源码和示例。
- 实现顺序必须是：Slint 原生组件与平台能力 > 项目已有组件组合 > 少量自定义组件 > 原生系统 API 手写实现。
- 菜单、二级菜单、弹层、托盘、焦点、窗口层级、边缘避让和键盘操作等能力，在确认 Slint 没有可用实现前，不得直接手写窗口或 Win32 交互逻辑。
- 原生组件只要能够满足核心交互，应优先采用；样式差异先通过属性、主题和组合方式调整，不能仅因默认外观不同就重新实现行为层。
- 只有原生组件明确无法满足产品行为、公开 API 不足或存在已确认缺陷时，才允许自定义实现。实现前需在代码注释或设计文档中简要记录限制、替代方案和自定义范围。
- 自定义组件应只补足缺失能力，不复制原生组件已经处理的定位、焦点、关闭策略、可访问性和跨平台行为。

## 验证要求

- UI 修改后至少执行 `cargo check` 或 `cargo build`。
- 涉及交互行为时，除编译检查外，还应验证鼠标、键盘、失焦关闭、屏幕边缘和不同缩放比例下的行为。
- 删除被原生组件替代的旧组件、回调、窗口管理代码和平台辅助函数，避免保留两套实现路径。

## 完成说明

- 最终说明必须区分官方已支持能力、项目已实现能力、当前设备已验证能力，以及尚未验证或规划中的能力。
- 涉及外部 API 时提供对应官方资料链接。
- 不得把“编译通过”描述为“功能验证通过”，也不得把接口存在描述为当前设备可用。
